# 一、项目背景与目标

## 1.1 行业与市场背景

近年来，生成式人工智能技术的快速发展使写作辅助工具从传统的语法检查与拼写纠错系统，逐步演变为能够参与内容生成、结构规划与知识总结的智能系统。以大语言模型为核心的新一代写作工具不仅能够根据提示生成完整段落，还可以对已有文本进行改写、扩展与风格调整，这一能力的提升正在改变用户对写作软件的预期，从"辅助工具"转向"协作伙伴"。

与通用写作领域相比，学术写作的数字化进程起步较早，但其工具生态长期围绕文献管理与排版展开，例如文献管理软件（Zotero、EndNote）与LaTeX编辑器（Overleaf）等。这些工具在文献组织与格式规范方面表现成熟，但在内容生成与知识整合方面能力有限。随着生成式AI进入学术场景，用户开始尝试将通用AI工具（ChatGPT、Claude等）引入论文写作流程，但现有产品形态尚未针对学术需求进行系统化整合。

根据2026年AI Index报告，全球90%以上高校学生已在学术场景中使用AI工具，但56%的AI生成引用存在错误，约20%为完全虚构。这一数据表明，学术写作场景亟需一个既能利用AI生成能力，又能保证引用准确性与学术规范性的专业化工具。

## 1.2 用户现有工作流程

在典型的学术写作过程中，学生与研究者通常经历选题调研→文献检索→资料整理→论文撰写→反复修改等多个阶段。以一位硕士研究生撰写毕业论文为例，其典型工作流如下：

- **选题阶段**：通过Google Scholar、CNKI等学术搜索引擎泛读领域论文，在浏览器中打开20-30个标签页，手动做笔记
- **文献管理阶段**：将PDF导入Zotero进行分类、标注与引用管理
- **撰写阶段**：切换到Word/Overleaf完成正文写作，通过Zotero插件插入参考文献
- **AI辅助阶段**：额外打开ChatGPT/Claude浏览器窗口，用于获取研究思路、理解复杂概念或润色段落
- **碎片整合**：在AI界面与本地写作环境之间频繁复制粘贴

## 1.3 当前流程的问题

多工具串联构成的学术写作流程在功能上虽然完整，但在体验层面呈现出明显的割裂特征：

1. **上下文切换频繁**：文献管理（Zotero/EndNote）→ AI交互（ChatGPT）→ 文档编辑（Word/Overleaf）三个系统之间不断跳转，打断思考连贯性
2. **AI缺乏文献上下文**：通用AI工具无法主动引用用户本地的论文资料，生成内容缺乏文献依据，引用准确率仅为40-60%
3. **引用管理碎片化**：引用条目散落在文献管理工具与文档插件之间，手动维护成本高且易出错
4. **知识资产无法沉淀**：每次写作过程中与AI的对话、修改记录、文献标注等分散在不同系统中，难以形成可复用的知识积累

## 1.4 项目目标

基于上述行业趋势与用户痛点，本项目的目标是构建一个面向学术写作场景的一体化智能工作平台，将文献管理、AI对话与论文编辑三类核心功能整合到统一界面中。核心目标包括：

- **SMART目标1**：用户完成一篇论文的平均工具切换次数从当前8-12次/小时降低至2次以下
- **SMART目标2**：AI生成内容的引用准确率≥90%（基于用户文献库），优于通用AI工具30-60%的水平
- **SMART目标3**：用户从文献导入到初稿完成的总写作时间缩短30%以上

## 2.3 竞品优劣势分析

**Paperpal**：优势在于投稿前检查覆盖30+参数和期刊合规性验证，Word/Overleaf插件集成成熟；劣势是缺乏文献管理能力和多轮对话式的AI交互。

**Jenni AI**：优势在于长文写作的实时续写（autocomplete）和2,600+引用格式支持，写作体验流畅；劣势是文献管理能力基础，不支持团队协作。

**沁言学术**：优势在于首创"左侧阅读管理、右侧引用生成"的交互模式，统一引用管理打通了文献到写作的数据链路；劣势是产品较新，生态和成熟度不及Paperpal。

**Research Companion**：优势在于本地优先的隐私策略（Ollama本地embedding），开源架构灵活；劣势是安装配置门槛高，非技术用户上手难度大。

## 2.4 市场空缺与机会点

综合以上竞品分析，市场空缺在于：**现有产品在文献管理、AI对话、文档编辑三者中最多覆盖两个维度，缺乏三合一深度整合的产品**。Paperpal擅长编辑但不善文献管理；Jenni AI擅长写作但文献能力基础；沁言学术有创新交互但编辑器能力有限。本项目瞄准"文献库驱动的AI写作"这一差异化定位，通过自建文献库+RAG技术实现"所写即所引"，填补市场空白。

# 三、产品定位与整体方案

## 3.1 产品定位

本产品定位为一款面向学生与科研人员的智能学术写作助手，核心差异化在于**"文献库驱动的AI写作"**（Literature-Grounded AI Writing）。与通用写作工具相比，系统关注学术场景下对文献引用、论证严谨性与内容可溯源性的特殊需求；与传统学术编辑平台相比，通过引入RAG（检索增强生成）技术使AI能够基于用户本地文献生成具有引用依据的内容。

目标用户：本科生（课程论文/毕业论文）、硕士研究生（期刊论文/学位论文）、初级科研人员（基金申请书/研究报告）。

## 3.2 核心价值主张

| 维度 | 现状（Before） | 解决方案（How） | 改变后（After） |
|------|---------------|----------------|----------------|
| 工具切换 | 在Zotero→ChatGPT→Word三工具间反复跳转 | 三栏一体化工作台 | 单一界面完成全部写作流程 |
| 引用可靠性 | AI生成引用40-70%为虚构 | 基于用户文献库的RAG生成 | 引用准确率≥90% |
| 知识沉淀 | 对话记录、文献标注散落各处 | 项目级知识库持续累积 | 写作资产可复用可迭代 |
| 写作效率 | 每次写作需重新搜集资料 | 文献库一次导入全文可复用 | 写作时间缩短30%+ |

## 3.3 产品整体架构

从功能结构上看，系统围绕文献管理、智能对话与文档编辑三大核心模块构建：

# 四、用户与使用场景

## 4.1 用户角色

在学术写作过程中，不同阶段与经验水平的用户在需求与使用方式上存在明显差异。本产品主要面向三类典型用户：

| 角色 | 核心痛点 | 关键需求 | 使用频率 |
|------|---------|---------|---------|
| 本科生 | 缺乏文献检索经验、论文结构组织能力弱 | AI辅助理解概念、生成初稿、格式规范 | 中频（每学期1-3篇） |
| 研究生 | 论证严谨性要求高、引用管理复杂 | 文献管理与引用自动化、RAG精准生成 | 高频（持续进行） |
| 科研人员 | 时间成本高、需要信息复用 | 快速检索历史文献、生成结构化草稿 | 高频（日常使用） |

## 4.2 核心使用场景

- **场景A-选题调研**：用户上传5-10篇领域核心论文 → AI基于文献生成领域概述与研究热点分析 → 用户确定研究方向
- **场景B-文献深度阅读**：用户批量导入论文PDF → 系统自动解析与向量化 → 用户通过自然语言提问获取跨文献的综合回答
- **场景C-论文撰写**：用户在编辑器中建立大纲 → 逐章节通过AI生成草稿（基于文献库）→ 手动修改完善
- **场景D-修改润色**：用户选中段落 → 请求AI改写/扩展 → 系统基于文献库生成改写版本并保留引用

## 4.3 用户旅程图

用户旅程各阶段与系统功能对应关系：

| 阶段 | 用户行为 | 系统能力 | 产出 |
|------|---------|---------|------|
| 项目创建 | 新建论文项目，设定研究主题 | 项目模板、领域知识图谱初始化 | 项目空间 |
| 文献导入 | 上传PDF，批量导入参考文献 | 自动解析元数据、文本提取、向量化 | 可检索的文献库 |
| 调研问答 | 输入研究问题，多轮对话 | RAG检索+LLM生成，附带引用来源 | 带引用的AI回答 |
| 大纲构建 | 请求生成论文大纲 | 基于文献主题聚类生成大纲建议 | 结构化大纲 |
| 正文撰写 | 逐章节写作，AI辅助生成段落 | 基于文献的段落生成、实时引用插入 | 论文正文 |
| 润色定稿 | 全文审阅、语言润色、格式检查 | AI润色、引用完整性检查、格式规范 | 终稿 |

# 五、产品功能设计

## 5.1 功能架构与优先级

使用MoSCoW法则对全部功能进行优先级划分：

| 优先级 | 占比 | 功能项 |
|--------|------|--------|
| **Must（必须有）** | 60% | 文献上传解析、语义检索、RAG问答、基础编辑器、导出 |
| **Should（应该有）** | 20% | 大纲生成、AI改写/续写、多格式导出、历史对话复用 |
| **Could（可以有）** | 15% | 团队协作、期刊模板、查重集成、语音输入 |
| **Won't（本次不做）** | 5% | 移动端App、视频讲解生成 |

## 5.2 文献库模块（Must）

### 5.2.1 功能描述

系统应支持用户上传学术论文文件（PDF为主），自动完成元数据解析、全文提取与向量化索引，以便后续语义检索与RAG问答使用。

### 5.2.2 用户故事

| ID | 用户故事 | 验收标准 | 优先级 |
|----|---------|---------|--------|
| US-01 | 作为研究生，我想要上传PDF论文，以便系统自动提取文献信息 | 1. 支持批量上传（最多20个PDF）<br>2. 自动提取标题、作者、摘要、年份<br>3. 上传后3秒内完成解析展示<br>4. 支持GB/T 7714、APA、MLA三种引用格式预览 | Must |
| US-02 | 作为研究者，我想要通过关键词或语义搜索文献库，以便快速定位相关论文 | 1. 支持标题/作者/关键词精确搜索<br>2. 支持自然语言语义搜索（如"讨论transformer架构的论文"）<br>3. 搜索结果按相关度排序<br>4. 每篇搜索结果展示标题、作者、年份、匹配片段 | Must |
| US-03 | 作为用户，我想要对文献进行标签分类，以便按主题组织知识库 | 1. 支持自定义标签创建<br>2. 支持标签颜色标记<br>3. 支持按标签筛选文献列表<br>4. 标签信息在项目间独立 | Should |
| US-04 | 作为用户，我想要在文献中查看AI生成的摘要，以便快速了解论文要点 | 1. 上传后自动生成200字中文摘要<br>2. 摘要基于全文内容生成<br>3. 摘要展示在文献详情面板 | Should |

### 5.2.3 功能需求列表

```
F1.  系统应支持用户上传PDF格式的文献文件（单个≤50MB），单次最多20个文件
F2.  系统应在上传后自动解析文献元数据（标题、作者、摘要、DOI、年份、期刊名）
F3.  系统应将解析后的文献全文进行向量化索引存储（使用text-embedding模型）
F4.  系统应允许用户查看已上传文献列表，支持排序（按上传时间/标题/年份）
F5.  系统应支持文献删除、标签编辑、详情查看操作
F6.  系统应支持文献全文预览（内嵌PDF阅读器）
F7.  系统应支持跨文献的语义检索，返回相关度排序的文献片段列表
```

## 5.3 AI对话模块（Must）

### 5.3.1 功能描述

系统应提供基于用户文献库的RAG（检索增强生成）智能问答能力，在生成回答时优先参考用户上传的文献资料，并提供引用来源溯源。

### 5.3.2 用户故事

| ID | 用户故事 | 验收标准 | 优先级 |
|----|---------|---------|--------|
| US-05 | 作为研究生，我想要向AI提问并得到基于我上传文献的回答，以便获取有依据的研究见解 | 1. 支持自然语言提问<br>2. AI回答需附带文献引用角标<br>3. 点击引用可跳转到对应文献片段<br>4. 回答生成时间≤5秒（正常网络） | Must |
| US-06 | 作为用户，我想要进行多轮对话，以便围绕同一主题深入探讨 | 1. 系统保留最近10轮对话上下文<br>2. 后续问题可引用前文内容<br>3. 支持"基于上文再详细解释"等指代性提问 | Must |
| US-07 | 作为用户，我想要将AI回答中的段落一键插入到编辑器中，以便快速构建论文草稿 | 1. 每条AI回答下方提供"插入到编辑器"按钮<br>2. 插入时保留引用角标<br>3. 自动在参考文献列表中添加对应条目<br>4. 支持插入到光标位置或替换选中文本 | Must |
| US-08 | 作为用户，我想要选择对话中使用的文献范围，以便聚焦特定主题 | 1. 支持选择"全部文献"/"指定文件夹"/"指定标签"三种范围<br>2. 切换范围后，后续回答仅基于选中文献生成<br>3. 范围选择控件放在对话输入框上方 | Should |
| US-09 | 作为用户，我想要保存有价值的对话记录，以便后续回顾 | 1. 对话历史自动保存<br>2. 支持按日期浏览历史对话<br>3. 支持对话记录导出为Markdown | Should |

### 5.3.3 功能需求列表

# 六、核心交互流程

### 6.1 主界面整体交互逻辑

系统采用三栏式一体化工作台布局，将文献库（左侧）、AI对话（中间）与论文编辑器（右侧）整合在同一页面中。用户进入系统后，即可在一个界面内完成资料查阅、问题咨询与论文撰写。三栏宽度可拖拽调整，适应不同使用场景（如写作时收窄文献栏、调研时展开文献栏）。

### 6.2 文献导入与建库流程

用户点击左侧上传按钮 → 选择本地PDF文件（支持多选） → 系统异步执行：①文档解析（元数据+全文提取）②文本分块③向量化索引 → 解析完成后文献条目实时出现在左侧列表 → 用户可继续提问或写作。单篇文献解析时间≤5秒（20MB以内PDF）。

### 6.3 基于文献的问答交互流程

用户在中间对话区输入问题 → 系统执行：①在文献库中语义检索Top-5相关片段②拼接检索结果与用户问题为prompt③调用LLM生成回答④在回答中标注引用角标 → 回答展示在对话区，引用可点击查看原文片段。

### 6.4 AI生成内容的插入流程

用户在对话区点击回答下方的"插入到编辑器" → 系统获取当前编辑器光标位置 → 将选中文本插入到对应位置 → 同步更新文末参考文献列表 → 如替换选中文本则执行替换。

### 6.5 已有文本的改写与优化流程

用户在编辑器中选中文本 → 点击浮窗中的"AI改写" → 弹出模式选择（改写/扩展/精简/学术化）→ AI基于文献库生成新版本 → 改写结果展示在对话区 → 用户预览后可采纳替换或重新生成。

### 6.6 交互流程闭环关系

文献导入为问答提供知识来源 → 问答结果为论文撰写提供素材 → 编辑器中的内容可反向触发新的提问或改写请求。三模块围绕统一数据层形成完整的"输入→处理→输出→反馈"闭环。

| 风险 | 分类 | 概率 | 影响 | 缓解措施 |
|------|------|------|------|----------|
| AI生成内容学术不端风险 | 老虎 | 高 | 高 | ①所有AI生成内容在界面明确标识；②提供引用溯源；③用户须知中强调审阅义务；④集成AI内容检测工具 |
| LLM引用幻觉 | 老虎 | 中 | 高 | ①RAG检索增强生成优先使用用户文献库；②检索结果不足时明确告知用户而非强行生成；③每轮回答标注引用来源 |
| 文献版权侵权 | 老虎 | 中 | 高 | ①用户上传文献仅用于个人RAG检索，不公开分享；②符合合理使用（Fair Use）原则；③用户协议明确版权责任 |
| 大模型API成本失控 | 纸老虎 | 中 | 中 | ①模型路由层按场景分配模型（简单问题用小模型）；②设置单用户日调用上限；③支持本地模型部署方案 |
| 数据安全泄露 | 老虎 | 低 | 极高 | ①全链路HTTPS加密；②数据库AES-256加密；③定期安全审计与渗透测试 |
| 竞品快速跟进 | 大象 | 中 | 中 | ①加速文献库与编辑器的深度整合，建立数据迁移壁垒；②快速迭代积累用户写作数据（脱敏后改进模型） |
| 用户写作数据隐私担忧 | 大象 | 中 | 中 | ①提供完全本地部署方案（Ollama+BGE）；②清晰的隐私政策；③支持数据导出与删除 |
| 各出版商PDF格式不统一 | 纸老虎 | 中 | 低 | ①支持多解析引擎（PyMuPDF+PdfPlumber+Grobid）；②解析失败时降级为纯文本提取；③建立错误反馈机制持续优化 |

## 13.3 AI伦理准则

1. **透明度**：所有AI生成内容在界面中有明确标识，不伪装为人工撰写
2. **可解释性**：AI回答附带引用来源，用户可以溯源验证
3. **人机协作**：AI定位为"协作伙伴"而非"替代作者"，用户始终保留最终修改权和署名权
4. **公平性**：模型不因用户学科、机构、国别产生系统性偏见
5. **隐私优先**：用户文献不上传至第三方模型训练集，支持本地部署方案
6. **持续监控**：建立AI输出质量评估机制，定期抽样检查引用准确率和内容质量

## 14.1 术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| RAG | Retrieval-Augmented Generation | 检索增强生成，在LLM生成回答前先检索外部知识库，将检索结果作为上下文输入模型，以减少幻觉 |
| 向量化/Embedding | Vectorization/Embedding | 将文本转换为高维向量表示，语义相似的文本在向量空间中距离更近 |
| 语义检索 | Semantic Search | 基于向量相似度（而非关键词匹配）的搜索方式，可理解查询意图 |
| 文本分块 | Text Chunking | 将长文档分割为固定长度或语义完整的文本片段，用于向量索引和RAG检索 |
| 提示词/Prompt | Prompt | 输入给语言模型的指令文本，引导模型生成期望的输出 |
| 引用幻觉 | Citation Hallucination | LLM生成不存在的论文引用或错误归属参考文献的现象 |
| MCP | Model Context Protocol | 模型上下文协议，Anthropic提出的AI模型与外部工具交互的标准协议 |
| MVP | Minimum Viable Product | 最小可行产品，包含核心功能的最小版本 |
| NPS | Net Promoter Score | 净推荐值，衡量用户推荐意愿的指标（0-10分） |
| SLA | Service Level Agreement | 服务等级协议，定义服务可用性与性能承诺 |
| MRR | Monthly Recurring Revenue | 月度经常性收入 |
| OCR | Optical Character Recognition | 光学字符识别，从扫描件/图片中提取文字 |
| DOI | Digital Object Identifier | 数字对象标识符，学术论文的唯一标识 |
| PMF | Product-Market Fit | 产品-市场匹配，产品满足市场需求的程度 |
| JTBD | Jobs To Be Done | 用户待完成的任务，需求分析框架 |
| OKR | Objectives and Key Results | 目标与关键成果管理法 |
