# 学术写作助手 · 技术选型决策记录（ADR）

## ADR-001：前端框架选型 —— React vs Vue

| 维度 | React 18 | Vue 3 | 结论 |
|------|----------|-------|------|
| 生态丰富度 | TipTap/ProseMirror编辑器、PDF.js均有React封装；MUI/Ant Design成熟度极高 | 编辑器与PDF组件生态较弱，需自行封装 | **React胜出** |
| TypeScript支持 | 模板即TSX，类型推导完善 | Composition API类型支持良好，但模板部分仍有gap | React略优 |
| 招聘市场 | 国内React岗位占比~60%，团队招募更容易 | ~30% | React胜出 |
| 学习成本 | Hooks心智模型有门槛 | 渐进式上手，对初级友好 | Vue胜出 |
| 大型项目经验 | Facebook/Meta内部验证10万+组件 | 超大型项目案例较少 | React胜出 |
| 性能 | 并发渲染（React 18）针对大组件树优化 | 静态提升+Block Tree，小应用性能更优 | 平手 |

**决策**：React 18 + TypeScript。核心原因是TipTap编辑器和PDF.js的React生态优势无法替代，这是产品三个核心模块中的两个。

## ADR-002：富文本编辑器 —— TipTap(ProseMirror) vs Quill vs Slate

| 维度 | TipTap | Quill | Slate | 结论 |
|------|--------|-------|-------|------|
| 插件化架构 | ProseMirror原生插件系统，可自定义Schema | Delta格式简单但扩展性弱 | 插件化设计良好但成熟度低 | TipTap胜出 |
| 协作编辑 | ProseMirror OT/CRDT生态成熟（Yjs集成） | 需自行实现 | 社区有方案但不稳定 | TipTap胜出 |
| 自定义节点 | 支持引用标注节点、AI提示节点等自定义渲染 | 不支持自定义节点类型 | 支持但API频繁变更 | TipTap胜出 |
| 学习曲线 | ProseMirror底层复杂，TipTap封装后大幅简化 | 简单 | 中等 | 平手 |
| 社区活跃度 | 35k+ GitHub Star，活跃维护 | 43k Star，维护放缓 | 29k Star | TipTap胜出 |
| 包大小 | ~80KB gzipped | ~40KB gzipped | ~50KB gzipped | Quill胜出 |

**决策**：TipTap（ProseMirror）。学术写作场景需要自定义引用标注节点和AI生成内容标识，这是Quill无法满足的硬需求。

## ADR-003：向量数据库 —— pgvector vs Milvus vs Qdrant

| 维度 | pgvector (PostgreSQL) | Milvus | Qdrant | 结论 |
|------|----------------------|--------|--------|------|
| MVP部署成本 | **零额外运维**，复用已有PG实例 | 需独立部署，至少2节点 | 需独立部署 | pgvector胜出 |
| 检索性能（10万向量） | <100ms，ANNS索引 | <10ms | <20ms | 平手（MVP无差异） |
| 扩展上限 | 单表~1亿向量 | 百亿级 | 十亿级 | 远期Milvus胜出 |
| 与PG的数据一致性 | **天然事务一致性**，文献表和向量表可联合查询 | 需自行保证双写 | 需自行保证双写 | pgvector胜出 |
| 过滤检索 | SQL WHERE原生支持元数据过滤 | 支持但需额外配置 | 支持 | pgvector胜出 |
| 运维技能要求 | 团队已有PG运维能力 | 需学习新运维栈 | 需学习新运维栈 | pgvector胜出 |

**决策**：**MVP阶段用pgvector**，10万级文献向量量完全满足；V2.0评估迁移至Milvus（当向量量超过100万时触发迁移评估）。渐进式架构，不提前过度设计。

## ADR-004：LLM模型选择 —— GPT-4o vs Claude 3.5 vs 本地Qwen3

| 维度 | GPT-4o | Claude 3.5 Sonnet | Qwen3 (本地部署) |
|------|--------|-------------------|------------------|
| 学术文本生成质量 | 优秀，逻辑连贯 | 优秀，论证更严谨 | 良好，中文场景有差距 |
| 引用准确率（RAG后） | ~85% | ~88% | ~75% |
| 长上下文处理 | 128K | 200K | 32K（Ollama运行） |
| 成本（1万字论文） | ~$0.15 | ~$0.12 | 电费仅$0.02 |
| 数据隐私 | 数据经OpenAI服务器 | 数据经Anthropic服务器 | **数据不出本机** |
| 响应延迟 | 3-8秒 | 2-6秒 | 15-30秒（消费级GPU） |

**决策**：**模型路由策略**，而非单一模型。
- 段落生成/大纲 → GPT-4o（创造性强）
- 文献摘要/逻辑推理 → Claude 3.5 Sonnet（论证严谨）
- 简单问答/改写 → GPT-4o-mini（成本优先）
- 隐私敏感场景 → 本地方案Qwen3+BGE-M3（Ollama部署，Phase 2交付）

## ADR-005：后端框架 —— FastAPI vs Django vs Express

| 维度 | FastAPI | Django | Express | 结论 |
|------|---------|--------|---------|-------|
| 异步支持 | 原生async/await | 4.0后支持但生态滞后 | 原生异步 | FastAPI胜出 |
| API文档自动生成 | OpenAPI(Swagger)开箱即用 | 需drf-spectacular | 需swagger-jsdoc | FastAPI胜出 |
| LLM生态整合 | LangChain/LlamaIndex原生Python | Python生态完整 | Python生态需桥接 | FastAPI/Django平手 |
| 数据库ORM | SQLAlchemy（灵活） | Django ORM（强约束） | Prisma/TypeORM | FastAPI灵活 |
| 学习成本 | 低 | 中 | 低 | FastAPI/Express平手 |
| 社区成熟度 | 72k Star，快速增长 | 老牌框架 | 老牌框架 | Django胜出 |

**决策**：FastAPI。核心原因是LLM调用链（LangChain）完全在Python生态中，FastAPI的异步模型天然适合I/O密集型的LLM API调用场景，且自动生成的OpenAPI文档可直接用于前端类型生成。

## ADR-006：文件存储 —— MinIO vs 阿里云OSS vs 本地文件系统

| 维度 | MinIO自部署 | 阿里云OSS | 本地文件系统 |
|------|------------|----------|-------------|
| MVP成本 | 服务器成本 | ¥0.12/GB/月 | 0 |
| S3兼容性 | 100%兼容 | 100%兼容 | 不支持 |
| 扩展性 | 水平扩展 | 无限 | 单机限制 |
| 数据迁移 | 可迁移到任何S3 | 厂商锁定风险 | 数据在本地 |
| 运维成本 | 需自己维护 | 0运维 | 0运维 |

**决策**：**MVP阶段用本地文件系统+定期备份**。用户量<1000时文献总量可控，单机SSD足够。V1.5评估接入OSS（触发条件：文献总量>10万篇或用户跨设备访问需求出现）。

## ADR-007：状态管理 —— Zustand vs Redux vs Jotai

| 维度 | Zustand | Redux Toolkit | Jotai |
|------|---------|---------------|-------|
| 包大小 | ~1KB | ~12KB | ~2KB |
| 学习成本 | 极低（hooks思维） | 中（reducer/action/thunk） | 低 |
| TypeScript友好 | 优秀 | 优秀 | 优秀 |
| 适合场景 | 中等复杂度应用 | 大型复杂应用 | 细粒度状态 |
| 三栏联动需求 | Module级状态天然适合 | 需精心设计Slice | 原子化状态更灵活但管理碎片化 |

**决策**：Zustand。学术写作助手的状态复杂度属于中等（3个模块、20-30个全局状态），Zustand的hooks接口足够且心智负担最小。如果未来引入团队协作导致状态复杂度剧增，再评估迁移Redux。

## 选型总结

| 层级 | 选型 | 核心决策逻辑 |
|------|------|-------------|
| 前端框架 | React 18 | 编辑器生态（TipTap）无法替代 |
| 富文本编辑器 | TipTap | 自定义引用节点是硬需求 |
| 状态管理 | Zustand | 中等复杂度，轻量够用 |
| 后端框架 | FastAPI | Python LLM生态+原生异步 |
| 向量数据库 | pgvector→Milvus | MVP零运维→规模化迁移 |
| LLM | 多模型路由 | 按场景选最优，不锁死单一供应商 |
| 文件存储 | 本地→OSS | 渐进式架构，按量触发升级 |
| 嵌入模型 | BGE-M3 | 中英双语最优，支持本地部署 |
| 部署 | Docker+K8s | 标准化交付，弹性扩缩 |

**每条决策的核心原则**：MVP够用不超配+预留迁移路径不锁死+优先复用团队现有技能栈。
